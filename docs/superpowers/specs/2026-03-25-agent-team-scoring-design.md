# X投稿フロー Agent Team 採点設計書

**日付:** 2026-03-25
**対象ファイル:** `score_posts.py`

## 概要

現在 Claude Haiku 1体で行っている投稿採点を、Sonnet 2体の Agent Team 構成に刷新する。採点精度を高め、キューに入る投稿の質を向上させる。

## アーキテクチャ

```
generate_posts.py（Claude Opus）
    ↓ 15件生成 → pending_review.txt
score_posts.py（Agent Team）
    ├── 採点役A（Claude Sonnet）
    │     既存 SCORING_CRITERIA をベースに
    │     scoring_context.json（バズ投稿・参考アカウント・トレンド）で微調整して採点
    └── 採点役B（Claude Sonnet）
          データなし、「コンサル転職志望の会社員」目線のみで採点
    ↓
    A・B 両方 7点以上 → PASS → pending_review.txt に保存
    どちらかNG → 生成役（Claude Opus）に差し戻し（最大2回）
        ├── フィードバック: 採点役A・Bの指摘を両方渡す
        └── 再生成 → 採点ループに戻る
    ↓
approve.py → queue.json → post_now.py
```

## 採点役の詳細

### 採点役A（基準採点）

- **モデル:** claude-sonnet-4-6
- **役割:** 品質基準に照らした客観的採点
- **ベース基準:** 既存の `SCORING_CRITERIA`（A〜Eの5軸、10点満点）
- **動的調整:** `scoring_context.json` を読み込み、直近のバズ投稿・参考アカウントのパターンを補足観点としてプロンプトに追加
- **出力（JSON）:** `score_A`〜`score_E`、`total`（0〜10）、`reason`（改善点50字以内）

### 採点役B（ターゲット目線）

- **モデル:** claude-sonnet-4-6
- **役割:** ターゲットの素直な反応をシミュレート
- **ターゲット定義:** コンサル転職・キャリアアップを目指す会社員
- **制約:** 調査データは渡さない（バズ基準に引っ張られないため）
- **出力（JSON）:** `total`（0〜10の整数）、`reason`（刺さったか・刺さらなかった理由50字以内）

## 合否判定

- 採点役Aの `total >= 7` かつ 採点役Bの `total >= 7` → PASS
- どちらか一方でも `total <= 6` → FAIL → 差し戻し

## リトライ（差し戻し）

- **差し戻しの実装:** `score_posts.py` 内で Claude Opus を直接呼び出して再生成する（`generate_posts.py` は呼ばない）
- **呼び出し順序:** 採点役A → 採点役B の直列呼び出し（並列化は不要）
- **フィードバック形式:** 採点役Aの `result["reason"]`（`reason_a` と変数名付与）とBの `result["reason"]`（`reason_b`）を以下の形式でプロンプトに連結する
  ```
  採点役A（品質基準）の指摘: {reason_a}
  採点役B（ターゲット目線）の指摘: {reason_b}
  ```
- **上限:** 最大2回
- 2回リトライ後も FAIL の場合はスキップ（キューに追加しない）

## pending_review.txt の扱い

既存フローを維持する。PASS した投稿は `pending_review.txt` に書き戻し、ユーザーが `approve.py` を実行してキューに追加する（自動でキューには入れない）。

## エラーハンドリング

- 採点役A または B の API 呼び出し失敗（ネットワークエラー・JSONパースエラー等）時: その投稿をスキップしてループを抜ける（差し戻しループには入れない）。ログには `[ERROR]` プレフィックスで採点エラーと明記し、採点スコア不足による FAIL と区別できるようにする。**設計意図:** APIエラー時に「採点エラー」という偽フィードバックで Opus を呼び出すのは API コストの無駄遣いになるため、即スキップとする
- 差し戻し中の Opus 呼び出し失敗時: その投稿をスキップしてログに出力する
- 採点役Aの `total` は `score_A`〜`score_E` の合計値で上書きして使用する（LLM の自己申告値は無視）
- 採点役Bの `total` は 0〜10 の整数であることをバリデーションし、範囲外の場合は FAIL として扱う。このバリデーションは初回・差し戻し後の再採点を含む全ラウンドに適用する
- 採点役Bは `total` と `reason` のみ出力する（`score_A`〜`score_E` の軸別スコアは出力・ログ表示しない）
